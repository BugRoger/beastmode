## Problem Statement

After PRD-1 (bm-d59e) delivers the structured task store alongside existing manifests, the system has two sources of truth — store.json for entity hierarchy/deps/queries and per-epic .manifest.json files for phase transitions/worktree/GitHub sync. This dual-store period introduces reconciliation overhead, drift risk, and cognitive load. The manifest module (store.ts, pure.ts, reconcile.ts) and its 43 consumer files must be rewritten to use the store as the sole persistence layer.

## Solution

Direct rewrite of 15 production files and 28 test files to replace all manifest reads/writes with store API calls. No adapter pattern — clean break. Store tests written first for confidence, then production code migrated file-by-file. Migration command (`beastmode store import`) converts existing .manifest.json files into store entities. After migration, manifest files and the manifest/ module are deleted.

This is PRD-2 of 2. Depends on PRD-1 (bm-d59e: Structured Task Store Foundation).

## User Stories

1. As a **pipeline orchestrator**, I want the pipeline machine to read epic/feature state from store.json instead of .manifest.json, so that phase transitions operate on the same data as dependency queries and ready-work computation.

2. As a **developer**, I want a single `store.json` file as the sole pipeline state, so that I don't need to understand two persistence formats or worry about drift between them.

3. As a **developer**, I want `beastmode store import` to migrate all existing manifests into the store, so that I can transition without losing pipeline state.

4. As a **watch loop**, I want to scan store entities directly instead of reading N manifest files from disk, so that epic discovery is a single JSON parse, not a directory scan.

5. As a **GitHub sync module**, I want to read epic/feature state from the store, so that issue enrichment and label sync use the same entity model as everything else.

6. As a **dashboard**, I want to render epic and feature state from store entities, so that the TUI shows the same data as `beastmode store` commands.

7. As a **developer**, I want the manifest/ module deleted after migration, so that there is exactly one state management module in the codebase.

8. As a **pipeline runner**, I want the 9-step execution flow to read/write the store directly, so that post-dispatch reconciliation operates on store entities instead of manifest JSON.

## Implementation Decisions

### Migration Strategy

**Store tests first, then production code migration.** Sequence:

1. Write comprehensive store integration tests covering all operations the pipeline needs (CRUD, phase transitions, feature status, worktree metadata, artifact references, dependency resolution)
2. Migrate production files one at a time, running tests after each
3. Delete manifest/ module after all consumers migrated

### GitHub State

GitHub issue numbers and body hashes are currently stored on manifest entities (`manifest.github`, `feature.github`). These move to a separate lightweight file that the GitHub sync module owns — not part of the store's entity model. The store models pipeline state; GitHub is a sync concern.

Alternative: store GitHub refs on entities as optional fields. Simpler but muddies the entity model.

Decision: GitHub refs stored as optional fields on entities (`github?: { issue: number; bodyHash?: string }`) for simplicity — the sync module reads/writes these fields. If this becomes problematic, extract to a separate file later.

### Pipeline Machine Integration

The XState v5 machine currently uses `EpicContext` (from `pipeline-machine/types.ts`) which mirrors `PipelineManifest`. After migration:

- `EpicContext` is replaced by the store's `Epic` type
- XState context reads from store on machine creation
- XState actions write to store (replacing in-memory accumulation + single `store.save()`)
- Guards read store state directly

The single-persist pattern changes: instead of accumulating state in memory and calling `store.save()` once at end of dispatch, the store is written after each significant state change. The store's file mutex ensures atomicity.

### Post-Dispatch Pipeline Changes

Current 9-step pipeline:
1. Worktree prepare → **unchanged**
2. Rebase → **unchanged**
3. HITL settings write → **unchanged**
4. Interactive dispatch → **unchanged**
5. Artifact collection → **unchanged** (still reads output.json)
6. Manifest reconciliation → **rewritten**: enriches store entities from output.json
7. Manifest advance → **rewritten**: advances epic status in store
8. GitHub sync → **rewritten**: reads from store entities
9. Worktree cleanup → **unchanged**

### Rename Ceremony Deletion

The 80-line `store.rename()` function is deleted entirely. With permanent hash-based IDs (`bm-xxxx`):
- Branch: `feature/bm-a3f8` (never renamed)
- Worktree: `.claude/worktrees/bm-a3f8` (never renamed)
- Entity ID: `bm-a3f8` (permanent)
- Display name: `epic.name` field (mutable, no file/branch implications)

The design phase sets `name` and `slug` after the PRD is approved. No rename of any git artifact.

### Migration Command

```bash
beastmode store import
```

Reads all `.beastmode/state/*.manifest.json` files and creates store entities:

1. For each manifest:
   a. Generate new `bm-xxxx` ID (or derive from `manifest.originId` if present)
   b. Create epic entity with all manifest fields mapped
   c. Create feature entities for each `manifest.features[]` entry
   d. Map `manifest.artifacts` to typed fields on entities
   e. Map `manifest.github` to optional github fields
2. Compute `depends_on` from `feature.wave` ordering (features in wave N+1 depend on all features in wave N within the same epic)
3. Validate: all entities created, all references resolve
4. Delete `.manifest.json` files on success

The migration is idempotent — running it twice on the same manifests produces the same store state (existing entities with matching origin are skipped).

### Backward Compatibility

During the transition:
- Old manifests are read-only after import (not updated by the pipeline)
- The `store.json` becomes the sole authority immediately after `beastmode store import`
- Skills continue to write artifacts with YAML frontmatter — the Stop hook generates output.json as before
- The output.json contract is unchanged — the CLI reads it and enriches store entities

## Testing Decisions

- Store integration tests written BEFORE production migration — comprehensive coverage of all operations the pipeline needs
- Each production file migration includes running the full test suite
- Migration command tested with real manifest fixtures from the codebase
- End-to-end test: design → plan → implement → validate → release cycle using store only
- Prior art: current `manifest-store.test.ts`, `manifest-pure.test.ts`, `pipeline-runner.test.ts` define the test scenarios to preserve

## Out of Scope

- Multi-clone sync (future backend)
- Permission enforcement on store operations
- SQLite/Dolt backends
- Task/step persistence (remains in .tasks.md)
- Dashboard visual changes (reads new data, same rendering)
- Status command visual changes (same)
- Watch loop behavioral changes (same orchestration logic, different data source)

## Further Notes

- The blast radius is significant: 15 production files, 28 test files. Store tests first mitigates risk.
- GitHub refs on entities is a pragmatic compromise — cleaner would be a separate sync-state file, but the added complexity isn't justified at current scale.
- The rename ceremony deletion is one of the highest-value outcomes — `store.rename()` is the most bug-prone code in the CLI.
- Wave-to-dependency conversion during import is lossy: wave ordering doesn't capture which specific features block which. The import creates conservative deps (all wave N+1 features depend on all wave N features). Manual refinement via `--add-dep` / `--rm-dep` can tighten this.

## Deferred Ideas

- Store change events (EventEmitter for store mutations — enables reactive dashboard updates)
- Store versioning / migration framework (for future schema changes)
- Separate GitHub sync-state file (extract from entity model)
- Store backup/restore commands
