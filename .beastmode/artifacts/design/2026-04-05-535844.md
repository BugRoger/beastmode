## Problem Statement

Dashboard epics display in arbitrary store insertion order. There is no date-based sorting, so the user has to visually scan the entire list to find recent work.

## Solution

Sort epics by `created_at` descending (newest first) in the shared data layer (`status-data.ts`). Active epics sort above done/cancelled epics, with each group sorted newest-first internally.

## User Stories

1. As a user viewing the dashboard, I want epics sorted newest-first by creation date, so that my most recent work is always at the top.
2. As a user viewing the dashboard, I want active epics grouped above done/cancelled epics, so that completed work doesn't clutter the top of the list.
3. As a user of the CLI status command, I want the same sort order as the dashboard, so that epic ordering is consistent across all views.

## Implementation Decisions

- Sort key is `created_at` timestamp on the store Epic entity
- Sort direction is descending (newest first)
- Two-tier grouping: active epics (design, plan, implement, validate, release) first, then inactive (done, cancelled) — each group sorted by `created_at` descending
- Sorting lives in `status-data.ts`, the shared data module consumed by dashboard and CLI — not in the view layer
- The "(all)" aggregate entry remains pinned at index 0 in EpicsPanel regardless of sort order — it is a UI construct, not a data entity

## Testing Decisions

- Unit test the sort function in status-data with mixed active/inactive epics and varying creation dates
- Verify "(all)" entry remains at index 0 after sorting
- Verify done/cancelled epics appear after all active epics
- Existing EpicsPanel tests reference a `STATUS_ORDER` constant — verify alignment or update

## Out of Scope

- User-configurable sort order
- Sort by updated_at or other fields
- Persisting sort preference
