## Problem Statement

Slug handling across the beastmode pipeline is brittle. Six resources are keyed by slug (git branch, worktree directory, worktree git metadata, manifest file, manifest content, design artifacts), managed by three separate owners (worktree.ts, manifest-store.ts, rename-slug.ts), with two independent rename paths (rename-slug.ts and store.save()'s internal rename logic), five potential persist calls per dispatch, a commit-message regex as the communication channel between the LLM and CLI, and inconsistent frontmatter field naming across phases (slug/epic/feature used interchangeably).

## Solution

Consolidate all slug lifecycle management into the manifest store. Introduce consistent terminology (slug = immutable hex, epic = derived human name, feature = sub-unit), standardize frontmatter across all phases, establish a single rename path (store.rename()), a single persist path (one store.save() per dispatch), and use output.json as the sole LLM-to-CLI communication channel.

## User Stories

1. As a user, I want `beastmode design` to create a worktree with a temp hex and rename it to a readable name after the session, so that branches and manifests are human-friendly
2. As a user, I want non-design phases to fail fast if the slug doesn't exist, so I don't get orphaned worktrees
3. As a user, I want to look up epics by either hex or name, so I can use whichever I remember
4. As a developer, I want a single rename method that handles all slug-keyed resources, so partial renames can't happen
5. As a developer, I want consistent frontmatter across all phases, so the stop hook and CLI don't need per-phase special cases
6. As a developer, I want one persist call per dispatch, so manifest state can't diverge between multiple writes
7. As a developer, I want collision resolution to use the birth hex as suffix, so naming is deterministic
8. As a user, I want the output.json to be findable by hex, so stale artifacts from previous sessions can't cause mismatches

## Implementation Decisions

### Terminology

- `slug`: immutable 6-character hex hash, assigned at worktree creation, never changes, used as the primary key for lookups
- `epic`: human-readable name derived by the design skill, used for filesystem naming (branches, worktree dirs, manifest files) after rename
- `feature`: sub-unit within an epic, created during plan phase, used in plan and implement artifacts

### Consistent Frontmatter

All phases use standardized fields:

| Phase | Always present | Phase-specific |
|-------|---------------|----------------|
| Design | `phase`, `slug`, `epic` | — |
| Plan | `phase`, `slug`, `epic`, `feature` | `wave` |
| Implement | `phase`, `slug`, `epic`, `feature` | `status` |
| Validate | `phase`, `slug`, `epic` | `status` |
| Release | `phase`, `slug`, `epic` | `bump` |

### Manifest Store Extensions

- `store.rename(projectRoot, hexSlug, epicName, worktreePath)`: single method that renames all slug-keyed resources. Prepare-then-execute strategy — validates all preconditions before mutating anything, falls back to fail-and-report on mid-execution failure.
- `store.find(identifier)`: resolves by either slug (hex) or epic (name), checking both manifest fields.
- `slugify()` and format validation (`[a-z0-9](?:[a-z0-9-]*[a-z0-9])?`) live in the store as the single chokepoint.
- Collision resolution: if epic name collides, use `<epic>-<hex>` suffix. Deterministic, one-shot, no `-v2` guessing.
- `originId` field persisted in manifest after rename for lineage tracking.

### Rename Sequence (store.rename)

1. Precondition checks: slugify + validate format, collision detection
2. Rename artifacts in worktree (.md + .output.json files, update frontmatter)
3. Commit renamed artifacts in worktree
4. Rename git branch (feature/hex → feature/epic)
5. Move worktree directory + repair git metadata
6. Rename manifest file
7. Update manifest content: set epic, set originId

Manifest is gitignored — only step 3 requires a git commit.

### Single Persist Path

- Machine persist action: updates context in memory only, no disk writes
- `store.save()`: write only, all rename logic removed
- `store.rename()`: renames files and updates fields, does not persist
- Post-dispatch: single `store.save()` at the end, after machine events + rename + GitHub sync mutations are all accumulated in memory
- `skipFinalPersist` flag deleted — one write path means no coordination needed

### Communication Channel

- output.json is the sole channel between LLM and CLI (generated by stop hook from frontmatter)
- Design skill writes `slug: <hex>`, `epic: <derived-name>` in frontmatter
- CLI finds output.json by hex match (unambiguous — hex is the artifact filename key)
- `resolveDesignSlug()` (commit message regex) deleted
- Stop hook (`generate-output.ts`): add `epic` field to design output, update plan phase to read `epic` field instead of legacy `epic` frontmatter key

### Non-Design Phase Behavior

- Slug must exist in store before worktree/branch operations
- Fail fast with error if not found — no worktree creation mid-transaction
- Design creates the slug and worktree; release ends the transaction
- Non-design phases are read-only with respect to slug identity

### Deletions

- `rename-slug.ts`: entire file deleted, logic absorbed into `store.rename()`
- `resolveDesignSlug()` in `post-dispatch.ts`: deleted
- `skipFinalPersist` flag in `post-dispatch.ts`: deleted
- Rename logic in `store.save()` (lines 184-198): deleted
- Machine persist action disk writes: removed

## Testing Decisions

- Unit tests for `store.rename()` covering: happy path, collision resolution with hex suffix, format validation rejection, precondition failure (missing branch/worktree), mid-execution failure and report
- Unit tests for `store.find()` covering: lookup by hex, lookup by epic name, ambiguous input
- Unit tests for `slugify()` and format validation
- Integration test: full design → post-dispatch → rename flow with actual git operations
- Prior art: existing tests in `cli/test/` for manifest-store and rename-slug

## Out of Scope

- Changing the watch loop or dashboard to use new terminology (downstream consumers can be updated incrementally)
- Pipeline machine (XState) refactoring beyond removing the persist action's disk writes
- GitHub sync changes beyond adapting to the new frontmatter fields
- Retroactive migration of existing artifacts to new frontmatter format

## Further Notes

- The `epic` field in plan/implement frontmatter currently uses the key `epic` already — this coincidentally aligns with the new terminology. The change is making `slug` consistently mean hex and adding `epic` to phases that only had `slug`.
- The `filenameMatchesEpic()` function in phase-output.ts will need to handle both hex-named files (pre-rename) and epic-named files (post-rename) during the design phase transition window.

## Deferred Ideas

- Inconsistent CLI command argument patterns across phases (plan takes epic, implement takes epic + feature, validate/release take epic) — could be standardized but is a separate UX concern
- `store.find()` could support fuzzy matching or partial hex prefix for convenience
- The `feature` sub-slug could benefit from the same validation/slugify treatment as the epic slug
