## Problem Statement

GitHub sync is fundamentally broken. Epic issue bodies contain no PRD content (Problem, Solution, User Stories, Decisions). Feature issue bodies contain no plan content (description, user story, what to build, acceptance criteria). Issue creation fails with `phase/undefined` because the sync engine reads nonexistent fields from the store type. Three field-name mismatches exist between the store schema and the sync input types. Feature titles use bare slugs instead of the `{epic}: {feature}` format. Artifact link URLs contain absolute local filesystem paths. There is no retry mechanism for GitHub API eventual consistency failures, and no reconciliation loop to fix issues that were created as stubs but never enriched.

## Solution

Fix the `syncGitHubForEpic` bridge layer to correctly map store entity fields to sync input types (`status` -> `phase`, flat phase path fields -> artifact resolution). Normalize all stored artifact paths to repo-relative form. Add retry queue metadata to the existing sync-refs file for tracking failed operations. Drain the retry queue on each watch loop tick (60s interval) with exponential backoff (5 retries). Implement full reconciliation that covers titles, bodies, labels, and project board status for any entity with pending/failed state. When sync-refs is empty, bootstrap it from the epic store in a state that triggers reconciliation for all existing entities.

## User Stories

1. As a project owner, I want epic issue bodies to contain the full PRD (Problem, Solution, User Stories, Implementation Decisions, Testing Decisions, Out of Scope), so that GitHub issues serve as readable project documentation.
2. As a project owner, I want feature issue bodies to contain the plan content (description, user story, what to build, acceptance criteria), so that feature issues describe what needs to be built.
3. As a project owner, I want issue creation to succeed on the first attempt by using the correct phase label from the store's `status` field, so that issues are created without `phase/undefined` errors.
4. As a project owner, I want failed GitHub API operations to be retried automatically with exponential backoff (5 retries), so that eventual consistency delays don't result in permanently broken issues.
5. As a project owner, I want a reconciliation pass on each watch loop tick that drains the retry queue, so that stubs get enriched and failed operations get retried without waiting for the next phase transition.
6. As a project owner, I want feature issue titles to use the `{epic}: {feature}` format consistently, so that features are identifiable in issue lists.
7. As a project owner, I want artifact link URLs to use repo-relative paths instead of absolute local filesystem paths, so that links are valid on GitHub.
8. As a project owner, I want sync-refs to bootstrap from the epic store when empty, populating entries in a state that triggers full reconciliation, so that existing issues with broken bodies get fixed automatically.

## Implementation Decisions

- The sync engine will read from store flat fields (`epic.status`, `epic.design`, `feature.plan`) instead of the nonexistent `epic.artifacts` map — no schema migration needed.
- `syncGitHubForEpic` will map `epicEntity.status` to `EpicSyncInput.phase` (fixing the `phase/undefined` bug).
- `readPrdSections` will accept a design artifact path string directly (from `epic.design`) instead of looking up `epic.artifacts["design"]`.
- Feature plan paths will be resolved by searching `.beastmode/artifacts/plan/` when the stored path is a bare filename.
- All artifact paths stored in the epic/feature entities will be normalized to repo-relative form during reconciliation.
- The retry queue will be stored as additional metadata in `github-sync.json` entries — a `pendingOps` array with operation type, retry count, and next-retry timestamp.
- Retry backoff: 5 attempts with exponential delay (1 tick, 2 ticks, 4 ticks, 8 ticks, 16 ticks). After 5 failures, the operation is marked as `failed` and logged.
- The watch loop's existing 60-second poll interval will include a `reconcileGitHubRetries()` call that processes all pending operations.
- Full reconciliation covers: issue title, issue body, labels (phase/status), project board status, and sub-issue linking.
- When `github-sync.json` is empty but the epic store has entities with GitHub-syncable state, the bootstrap process creates sync-ref entries with `bodyHash: undefined` for each entity — the hash mismatch triggers body reconciliation on the next sync pass.
- Stubs created by early-issues will immediately have their entity ID added to the retry queue with a `bodyEnrich` operation, ensuring the next tick attempts enrichment.

## Testing Decisions

- Unit tests for the `syncGitHubForEpic` field mapping (status -> phase, design path resolution, feature plan path resolution).
- Unit tests for retry queue operations (add, drain, backoff calculation, max-retry expiry).
- Unit tests for sync-refs bootstrap logic (empty sync-refs + populated store -> bootstrap entries).
- Integration test: create epic stub, run reconciliation, verify body contains PRD sections.
- Prior art: existing tests in `cli/src/store/json-file-store.test.ts` for store operations, `cli/src/github/` for sync patterns.

## Out of Scope

- Reverse sync (reading GitHub state to update the local store) — remains one-way mirror.
- GitHub webhook integration — reconciliation is poll-based via watch loop tick.
- Historical artifact path migration in the store — only new artifacts get normalized paths; existing paths are resolved at sync time.
- Projects V2 board field configuration changes.

## Further Notes

- The `phase/undefined` bug means every epic created since the store migration has failed its initial GitHub sync. The retry-on-next-phase behavior partially masked this since the second dispatch would succeed if the epic already existed in sync-refs.
- The feature title bug in early-issues (`feature.slug` vs `featureTitle(epicName, feature.slug)`) is also covered by full reconciliation — the enrichment pass updates titles as part of its sweep.
- TypeScript didn't catch the field mismatches because the store's `getEpic()` return type flows through loosely typed intermediate objects. Tightening the `syncGitHubForEpic` parameter types would prevent recurrence.

## Deferred Ideas

- Type-safe store accessor that prevents `any`-typed field access (would prevent the `phase`/`status` mismatch class of bugs).
- GitHub webhook listener for real-time sync instead of poll-based reconciliation.
- Artifact storage redesign — centralized artifact registry instead of scattered path fields on entities.
