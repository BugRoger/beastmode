## Problem Statement

The five beastmode checkpoint phases have inconsistent artifact writing, frontmatter fields, STOP termination, and handoff format. This breaks the Stop hook's ability to reliably generate output.json for the CLI post-dispatch pipeline. Specifically: release has no STOP, implement writes no artifact on clean runs, handoff commands mix slash commands and CLI commands, and frontmatter fields vary arbitrarily between phases.

## Solution

Normalize all five checkpoint phases to follow a uniform contract:
1. Always write an artifact with standardized frontmatter (`phase`, `slug`, `status` at minimum)
2. Always end with `STOP. No additional output.`
3. Always use `beastmode <phase> <slug>` for handoff commands

## User Stories

1. As the CLI post-dispatch pipeline, I want every phase to produce an artifact with frontmatter so that the Stop hook can reliably generate output.json regardless of phase outcome.
2. As the watch loop, I want every phase to end with STOP so that session termination is unambiguous and the Stop hook fires consistently.
3. As a user reading handoff output, I want all phases to use the same `beastmode <phase> <slug>` format so that I can copy-paste the next command without thinking about whether it's a slash command or CLI command.
4. As the Stop hook script, I want all artifacts to have `phase`, `slug`, and `status` fields so that output.json generation doesn't need per-phase special-casing for missing fields.

## Implementation Decisions

- **Minimum frontmatter contract**: All phases write `phase`, `slug`, `status` in artifact frontmatter. This is the universal floor.
- **Phase-specific extras**: `bump` (release), `epic`/`feature` (plan/implement) are extras above the minimum â€” not required by the contract.
- **Design frontmatter**: Add `status: completed` to design checkpoint template (currently only has `phase` and `slug`).
- **Release frontmatter**: Add `status: completed` to release artifact frontmatter (currently has `phase`, `slug`, `bump` but no `status`).
- **Release STOP**: Append `STOP. No additional output.` after "Release complete." and the marketplace update suggestions.
- **Implement artifact**: Always write a completion artifact with frontmatter, even when there are no deviations. Deviation details are appended to this artifact when they exist.
- **Handoff format**: Standardize on `beastmode <phase> <slug>` everywhere. Fix implement's `Next: /validate` to `Next: beastmode validate <slug>`. Plan continues to list multiple implement commands (fan-out). Release ends with "Release complete." (terminal phase, no next step).
- **Frontmatter fields**: Plan and implement additionally keep `epic` and `feature` for fan-out. `slug` in plan/implement is the composite `<epic>-<feature>`.
- **Validate FAIL path**: Keep existing behavior (STOP without commit on failure, no artifact written) but normalize STOP text to `STOP. No additional output.` for consistency.
- **Validate FAIL STOP**: Change from `STOP â€” do not proceed to commit` to `STOP. No additional output.` â€” the semantic difference (no commit, no artifact) is structural, not textual.
