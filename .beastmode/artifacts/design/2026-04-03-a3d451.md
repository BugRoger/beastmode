## Problem Statement

The task runner (`skills/task-runner.md`) is a 90-line state machine that parses phase headings into TodoWrite tasks, manages lazy expansion, retries, parent-child completion tracking, and a blocking/deadlock detection loop. All of this machinery exists to execute 4 sequential phases that are inherently sequential. It adds overhead without value — every skill invocation pays the parsing and tracking cost for a state machine that never branches.

## Solution

Remove the task runner entirely. Flatten all phase files and reference docs into a single SKILL.md per skill. Phases become `## Phase N` headings executed top-to-bottom. Use Claude Code's built-in TaskCreate/TaskUpdate for lightweight progress tracking. Delete all task-runner context documentation.

## User Stories

1. As a skill author, I want each skill to be a single self-contained SKILL.md file, so that I can read and edit the entire skill in one place without navigating phase subdirectories.
2. As a skill consumer, I want phases to execute directly without a parsing/tracking middleman, so that skill invocations are faster and simpler.
3. As a skill author, I want HARD-GATE blocks to contain only behavioral constraints (not task-runner bootstrapping), so that the enforcement mechanism is focused and readable.
4. As a contributor, I want stale task-runner references removed from context docs, so that the knowledge hierarchy accurately reflects the current system.
5. As a pipeline operator, I want the init skeleton to remain clean of task-runner references, so that new project installs don't get stale documentation.

## Implementation Decisions

- **Progress tracking via TaskCreate/TaskUpdate**: Replace TodoWrite-based tracking with Claude Code's built-in task tools. Each phase gets one task entry. Simpler, same visibility.
- **HARD-GATE blocks preserved**: Remove the `Execute @../task-runner.md now` and `Your FIRST tool call MUST be TodoWrite` lines. Keep phase-specific constraint lines (e.g., "No implementation until PRD is approved").
- **Full flatten into SKILL.md**: Inline all phase file content as `## Phase N` headings. Inline all reference file content. Delete `phases/` and `references/` subdirectories per skill. One file per skill, nothing else.
- **No `@` imports in SKILL.md**: Replace agent file references (`@../../agents/implement-implementer.md`, `@../../agents/common-researcher.md`) with plain text instructions describing the agent role. SKILL.md is fully self-contained.
- **Delete `skills/task-runner.md`**: Dead code after migration.
- **5 skills affected**: design, plan, implement, validate, release. The `beastmode` skill (init/ideas/setup-github) uses subcommands, not phases — unaffected.
- **Estimated SKILL.md sizes**: design ~306 lines, plan ~329 lines, implement ~560 lines, validate ~147 lines, release ~306 lines. All within acceptable range.
- **Agent files remain separate**: Agent prompt files in `agents/` are shared across skills and stay as standalone files. Skills reference them by description, not by `@` import path.
- **Agent reference format**: Name + one-sentence role description (e.g., "Spawn a general-purpose agent as the implementer. It receives the task spec and produces code changes."). No file paths, no full prompt inlining.
- **HARD-GATE entry point**: Behavioral constraint only, no TaskCreate instruction. Phases are headings, execution is implicit top-to-bottom.
- **DESIGN.md Task Runner section**: Delete entirely. TaskCreate/TaskUpdate is standard Claude Code tooling — no beastmode-specific convention needed.

## Testing Decisions

- Invoke each of the 5 skills (`/design`, `/plan`, `/implement`, `/validate`, `/release`) and verify they execute phases in order without errors
- Verify HARD-GATE constraints still trigger (e.g., design still blocks implementation until PRD approval)
- Verify TaskCreate/TaskUpdate progress tracking appears correctly during skill execution
- Verify no dangling references to deleted files (`task-runner.md`, `phases/*.md`, `references/*.md`)
- Grep the full skill tree for `task-runner`, `TodoWrite`, `@../task-runner`, and `phases/` to confirm no stale references

## Out of Scope

- Changing the 4-phase structure (prime/execute/validate/checkpoint) — phases stay, just inline
- Modifying the `beastmode` skill (init/ideas/setup-github) — different structure, not phase-based
- Changing agent files in `agents/` — they stay as-is
- Modifying historical design/research artifacts about the task runner — left as historical record
- Changing the CLI or pipeline orchestration — skill-level change only
